Let's dispel some myths.

Myth #1

Only professional developers or maintainers can contribute to open source projects.

Open source projects are built by the community, for the community, and you would be hard-pressed to find a project that doesn't welcome help of some form. People of all levels can contribute to open source projects.

Myth #2

Only experts can contribute something meaningful.

There is room for people to contribute in a variety of ways. As a non-expert, your experience with the project will be different than that of a seasoned maintainer, and this is tremendously helpful because the experts will have a lot more trouble understanding what matters to people who aren't experts.

Think about it – what's more meaningful to a newcomer: better documentation about getting started or a new advanced feature?

Find a project to work on

Finding something to work on is often the hardest part of contributing to open source, especially, if you are only considering code contributions as your options. However, there are many other ways to contribute e.g., improving documentation, making tutorials, building community, and reporting bugs.

Evaluate potential projects for suitability

1. Software license

2. Recent activity

3. Codebase size

4. Tech stack

5. Repository access

  • Which license is the software distributed with? Your employer may have restrictions on your participation depending on the license.
  • Is the project still active? Has it been a long time since the project was last updated/released? Are people still using it? If not, the project may have been abandoned – try to find something else.
  • How big is the codebase? While there will likely be more opportunities for contributions in a larger codebase, it may be harder to get a foothold if you are newer to the codebase.
  • Is the project built on top of another project that you are familiar with? If so, you can use that knowledge to contribute even if you are less familiar with this project's codebase. Example: Seaborn is built on top of Matplotlib, so if you know Matplotlib, you can tackle some issues in Seaborn.
  • Can you find where the project's repository is hosted? Is it public? If not, then this isn't an option.

Identify a way to contribute

Hopefully, some of the projects you brainstormed are suitable. Now, you will have to find something within one of those projects to work on. Here are five strategies that I have used.

1. Participate in sprints

  • Meet and work with some of the maintainers of a project.
  • Often, no prior experience with the project is necessary.
  • Maintainers curate a set of issues for the sprint and help people contribute.
  • Tremendous opportunity to learn from maintainers.

2. Contribute examples to the documentation

  • Every project needs good documentation.
  • Writing documentation is time-intensive.
  • Contributing examples to the helps you learn about the functionality and helps others learn from the improved documentation.
Screenshot of NumPy's Polynomial.degree() documentation page with examples from the merged pull request. (Source: Stefanie Molin)
  • Documentation paradox: projects need good documentation, but creating it takes a lot of time.
  • Thorough documentation may be deprioritized by maintainers in favor moving the project along.
  • As someone from the outside, you may be in a better position to provide an example because you know what is misinterpreted from a user's perspective.
  • Experts may have a harder time understanding what newcomers will need.

3. Browse open issues for ideas

  • Look for tags such as beginner, easy, or good first issue.
  • Don't disregard old issues, but make sure to check those changes are still desired.
  • Search for meta-issues.
  • Make sure to indicate that you are working on something by commenting on the issue.
Screenshot of open issues tagged "good first issue" in the scikit-learn repository as of July 24, 2023 11:44AM. (Source: Stefanie Molin)

Sifting through a project's issues can be overwhelming due to the volume of issues, but knowing how to search helps significantly:

  • Look for tags such as beginner, easy, or good first issue.
  • Try to find issues with few (or no) comments as these are likely just the description of the task. In GitHub, you can see this from the listing of issues, which saves you time compared to opening each issue individually.
  • Older issues may have less action, but may still be relevant. Be sure to ask the maintainers.
  • Check for meta-issues. These are issues that list a set of desired changes that should be performed on multiple areas of the codebase. Often, this means you have a template to follow. These are common for sprints.

Comment on the issue if you are interested in working on it. You can also ask the maintainers for clarifications here.

4. Identify and fix a bug

  • Is there a typo or omission in the documentation?
  • Does the software not work as expected?
  • Put in a bug report and volunteer to fix it if the maintainers agree it should be addressed.
  • Feel free to ask for help from the maintainers (you are helping them out by contributing).
Screenshot of Bug Report issue template for the pandas GitHub repository as of July 24, 2023 11:52AM. (Source: Stefanie Molin)

5. Propose and implement a new feature

  • What are the pain points of using the software? Is there something that could be done to address them?
  • Is there a feature missing?
  • Share your idea with the maintainers and work on it if it is accepted.
  • Don't take things personally.
  • Always be polite.
Screenshot of issue #450 in the numpydoc GitHub repository, which proposes incorporating a pre-commit hook for numpydoc validation. (Source: Stefanie Molin)

No contribution is too small.

I presented the five strategies in order of the level of experience you will likely need with a given project (least to most). There is no "best strategy" – tailor your contribution to your comfort level.

Making your contribution

Once you have found something that you would like to work on, make sure you claim it (e.g., by commenting on the issue). This keeps the maintainers in the loop and prevents duplication of efforts.

Tips for success ‚ú®

  • Let the maintainers (and the world) know what you are working on.
  • Follow any setup instructions the project provides.
  • New to development? Start with documentation-based contributions, while you get comfortable working with tools like Git and GitHub.
  • If you can, attend a sprint or perhaps a local meetup for your preferred programming language.
  • Be polite and patient in all interactions.

  • Make sure to follow any setup instructions the project provides for contributors. This may be in the documentation or in a contributing guide in the repository. If the instructions don't work for you, you can also fix that.
  • If you are completely new to development, start with documentation-based contributions, while you get comfortable working with tools like Git and GitHub. If you can, attend a sprint or perhaps a local meetup for your preferred programming language.
  • Remember to be kind and patient in all your interactions with a project. Many maintainers are doing this outside of their regular work – it might take some time for them to review everything.

Keep up the momentum üí™

  • Look for knowledge-sharing opportunities across projects.
  • Help others contribute.
  • Know that your dedication will not go unnoticed.

Be sure to keep your eyes open for opportunities of cross-pollination: something you are exposed to while working on one project may be something that another project wants to implement.

Take a look at the open PRs. If you see something needs to be fixed, feel free to comment on it. This is also a great way to get noticed by the maintainers if you are interested in joining them.

Your dedication will not go unnoticed: sticking with it can lead to an invitation to become a maintainer.
